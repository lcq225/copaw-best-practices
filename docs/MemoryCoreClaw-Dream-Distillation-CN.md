# MemoryCoreClaw 最佳实践：梦境蒸馏机制

> **状态**：生产就绪 (v3.3)  
> **作者**：Mr. Lee & MemoryCoreClaw 团队  
> **最后更新**：2026-04-18

## 📘 概述

**梦境蒸馏（Dream Distillation）** 是 `MemoryCoreClaw` 记忆系统的自动化维护管道。它将碎片化的短期日志转化为结构化、高价值的长期知识，确保记忆数据库保持精简、无冲突且语义丰富。

它像一个后台"清洁工"，每周自动运行，防止记忆膨胀并自动提炼历史交互中的行为规则。

## 🏗️ 架构：五阶段流水线

蒸馏过程分为五个独立阶段，按顺序执行，每个阶段都具备故障保护机制。

### 阶段 1：审计（Dry Run）
- **目标**：扫描数据库中的潜在冲突、重复项和低重要性条目，不执行实际更改。
- **机制**：使用 Jaccard 相似度进行快速关键词匹配，使用 Embedding 相似度进行语义分析。
- **输出**：生成健康报告（仅日志记录）。

### 阶段 2：智能清洗
- **目标**：去除噪声，同时保护关键规则。
- **操作**：
  - **衰减**：对最近未访问的 `event/log` 条目应用重要性衰减因子（如 `importance * 0.95`）。
  - **豁免**：关键类别（如 `rule`、`path`、`preference`）**免疫衰减**。
  - **阈值**：重要性降至地板值（如 `0.3`）以下的条目将被软删除。

### 阶段 3：核心提炼（LLM 驱动）
- **目标**：从原始数据中提炼可操作的洞察。
- **机制**：
  - 使用 LLM（智能路由：云端失败降级本地 Ollama）分析高排名记忆。
  - 提取高层行为规则（如"删除文件前必须确认"）。
- **故障保护**：如果 LLM 服务超时，脚本将保留上一版规则集，避免崩溃或生成垃圾数据。

### 阶段 4：MEMORY.md 原子更新
- **目标**：将蒸馏内容同步回人类可读的 `MEMORY.md` 文件。
- **核心创新：基于锚点的更新（`<!-- START_AUTO -->`）**
  - 脚本识别 `MEMORY.md` 中的特定自动管理区块。
  - **去重**：写入前检查新内容是否已存在于文件的手动部分，防止重复。
  - **多锚点熔断**：如果文件损坏（如找到多个锚点），脚本**立即中止**，防止进一步损坏。
  - **原子写入**：使用 `.tmp` 文件和 `os.replace` 确保即使进程被终止也能保持文件完整性。

### 阶段 5：归档与通知
- **目标**：长期存储管理。
- **操作**：
  - 将超过 30 天的每日笔记移动到 `archive/` 目录，按月组织。
  - 尝试更新 `config.json` 搜索路径（带严格验证）。
  - 用执行摘要更新 `HEARTBEAT.md`。

## 🛡️ 生产级安全保障

### 1. 并发保护（WAL 模式）
- 脚本在 SQLite 数据库上强制启用 `PRAGMA journal_mode=WAL` 和 `PRAGMA busy_timeout=8000`。
- 防止 Agent 同时读写记忆时出现"Database is locked"错误。

### 2. 智能防抖（Anti-Spam）
- 与 Heartbeat 系统集成。
- 使用 `.last_run_distiller` 时间戳文件确保完整管道**每周仅运行一次**，即使心跳触发多次。

### 3. LLM 路由与降级
- **主路由**：内部云 API（高性能）。
- **降级路由**：本地 Ollama（离线能力）。
- **网络**：内部请求禁用代理，防止超时。

## ⚙️ 配置示例

要在 QwenPaw/CoPaw 环境中启用梦境蒸馏：

### 1. Heartbeat 配置
编辑 `HEARTBEAT.md` 添加每周触发器：

```markdown
## 梦境蒸馏器（每周 Heartbeat）
- **时间窗口**：10:00 - 13:00。
- **防抖检查**：读取 `skills/memorycoreclaw/.last_run_distiller`。如果 7 天内更新过，则跳过。
- **命令**：`python skills/memorycoreclaw/scripts/dream_distiller.py`
- **模式**：静默运行（仅记录到控制台）。
```

### 2. 文件结构
确保存在以下结构：
```text
skills/memorycoreclaw/
├── scripts/
│   └── dream_distiller.py   # 主可执行脚本
├── .last_run_distiller      # 防抖状态文件
└── SKILL.md
```

## 📝 经验教训
1. **不要盲目修改配置**：永远不要在备份和验证前修改 `config.json`。使用 try-catch 块。
2. **防止文件膨胀**：在追加到 `MEMORY.md` 前始终实现去重。我们修复了一个锚点被重复 180 次的 bug！
3. **静默是金**：后台任务不应打扰用户。使用静默模式和摘要日志。

## ❓ 常见问题

### 自己的项目也需要用 PR 吗？
**是的，强烈建议。** 即使是个人项目，PR 也是最佳实践：
- ✅ **保留审查历史**：每次变更都有记录可查
- ✅ **触发 CI/CD 检查**：自动运行测试和验证
- ✅ **文档化变更**：PR 描述本身就是变更日志
- ✅ **未来协作友好**：如果将来有合作者，PR 流程无缝衔接
- ✅ **支持功能分支**：主分支永远保持可发布状态

### 多久运行一次合适？
推荐**每周一次**（如周一 11:00 AM）。通过 Heartbeat 的 4 小时间隔 + 防抖检查，可以确保：
- 不会漏跑（有心跳兜底）
- 不会重复跑（防抖机制）
- 资源消耗最小化
